# POST /api/games 测试设计

## 被测对象

- 接口：`POST /api/games`
- 作用：创建一个新的内存 2048 游戏实例，并返回该实例的 `id` 与初始 `Game2048State`

## 输入因子

| 因子 | 说明 | 等价类 / 边界值 |
| --- | --- | --- |
| 请求体 | 当前接口忽略请求体内容，Cucumber 场景统一传 `{}` | 1. 空对象 `{}` |
| 下一次创建游戏的 `id` 来源 | 生产路径使用默认生成器；e2e 测试通过 JFactory Spec `下一个新游戏标识` 预设下一次创建时要使用的 `id` | 1. 未预设，走默认随机生成 2. 已预设，使用场景显式指定的 `id` |
| 预设 `id` 内容 | 仅在使用 `下一个新游戏标识` Spec 时有意义，需要是一个明确的非空字符串 | 1. 合法固定值（如 `ui-created-game`） |

## 输出因子

| 因子 | 说明 |
| --- | --- |
| HTTP 状态码 | 成功时返回 `200` |
| 返回体 `body.json.id` | 默认路径返回随机 32 位十六进制字符串；测试预设路径返回场景指定值 |
| 返回体 `body.json.state` | 新局初始状态：分数为 0、棋盘 4x4、无胜负、可继续移动 |
| 内存中的游戏缓存 | 新建游戏实例应写入内存缓存，并使用返回的 `id` 作为 key |
| SQLite 存档/排行榜 | 创建新游戏时不应写入 `SavedGames` 或 `LeaderboardEntries` |

## 关键路径

```text
[收到 POST /api/games]
  ↓
{测试是否预设了下一次创建 id?}
  ├── N → [使用默认 id 生成器]
  └── Y → [消费预设 id]
           ↓
      [创建新的 Game2048 实例]
           ↓
      [写入内存缓存 games[id]]
           ↓
      [返回 200 + { id, 初始 state }]
```

## 用例矩阵

| 用例名 | 请求体 | id 来源 | 预设 id | 预期返回 | 预期副作用 |
| --- | --- | --- | --- | --- | --- |
| `create_game_with_default_generated_id` | `{}` | 默认生成器 | 不适用 | `200` + `id` 匹配 32 位十六进制 + 初始新局状态 | 内存缓存新增 1 个新游戏；数据库无新增存档/排行榜 |
| `create_game_with_cucumber_controlled_id` | `{}` | 测试预设 | `ui-created-game` | `200` + `id='ui-created-game'` + 初始新局状态 | 先由 JFactory Spec `下一个新游戏标识` 准备 next id；随后内存缓存新增 1 个新游戏，key 为 `ui-created-game`；数据库无新增存档/排行榜 |

## 覆盖性检查

1. 代码路径覆盖：默认 id 生成路径、测试预设 id 路径都已覆盖
2. 输入取值覆盖：`id` 来源的两类取值都至少在一个场景中出现
3. 判断逻辑覆盖：
   - 是否存在测试预设 id：覆盖有 / 无两种分支
   - 创建新局后的返回契约：覆盖 `id` 与完整初始 `Game2048State`
   - 创建接口的无持久化副作用：通过设计约束要求保持 `SavedGames` / `LeaderboardEntries` 不变
